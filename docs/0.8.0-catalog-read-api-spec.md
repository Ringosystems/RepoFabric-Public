# Catalog-read API spec (RepoFabric#2, M6 bolt-on)

Operated by RingoSystems Heavy Industries. Implementation spec for the read-only
catalog-presence + projection-export API that ConfigFabric's deep prerequisite
resolver consumes (Ringosystems/RepoFabric#2; contract on Ringosystems/ConfigFabric#1).
Branch: `0.8.0-catalog-read-api`.

## Scope split (why two PRs)

The catalog-read API has one hard external dependency: the **constraint/range
grammar** (decision-register Q2), which ConfigFabric must freeze
(Ringosystems/ConfigFabric#3) before the satisfaction check can be built. So:

- **PR1 (this branch, buildable now, grammar-independent):** presence
  point-query (existence + promotion stage + virtual-repo coherence) and the
  bulk enumeration / projection-export with an incremental `since` cursor. These
  need only exact `(repo_id, app_id, version)` matching, which the existing
  catalog already supports.
- **PR2 (gated on Q2):** the `satisfied` / `satisfyingVersions` constraint
  evaluation (FR-2, FR-3), built on the now-unified `ConvertTo-RfVersionSortKey`
  comparator and the pinned npm-style grammar. RepoFabric returns the verdict so
  ConfigFabric never recomputes satisfaction client-side.

## Data model (grounded in the current schema)

- **`repo_catalog`** (migration `012`, extended per-repo in `020`): `package_id`,
  `package_name`, `publisher`, `latest_version`, `version_count`,
  `versions_json` (all versions, descending), `first_seen_at`, `last_seen_at`,
  plus `repo_id` (index `ix_repo_catalog_repo`). Presence of a specific version
  is a membership test against `versions_json` for the `(repo_id, package_id)`
  row.
  - **First task / verify:** confirm whether `020` rebuilt `repo_catalog` to a
    composite `(repo_id, package_id)` primary key (the `012` PK was
    `package_id` alone). The point-query and enumeration both assume per-repo
    rows; if the PK is still global, that rebuild is a PR1 prerequisite.
- **Promotion stage:** there is no stage column today. PR1 adds
  `virtual_repos.stage` as a nullable `CHECK`-constrained column
  (`main` | `dev` | `test` | `departmental`) in the **next free migration
  number `033`** (`032` is taken by the source_fabric migration). When null the
  endpoint returns the `repo_id` as the stage (slug passthrough), per Q10.
- **Version comparison:** the single authority is `ConvertTo-RfVersionSortKey`
  (already converged in this milestone). Ordering in enumeration and the PR2
  satisfaction check both use it.

## Endpoints (PR1)

All under the publisher bridge (loopback, Bearer-gated by the listener, the same
posture as `POST /api/audit/events`). HTTPS at the edge via NPM.

### Presence point-query (FR-1, FR-4, FR-11, FR-12)
```
GET /api/v1/catalog/apps/{appId}/presence?repoId={slug}&version={v}
-> 200 { present: bool, repoExists: bool, repoId, appId, version,
         promotionStage, coherent: bool, asOf: <watermark> }
-> 404 only when appId is malformed; unknown repo -> 200 { repoExists:false, present:false }
```
- `app_id` matched case-insensitively, folded at the seam (Q3).
- `coherent` is false when the version is present only in a sibling slug, not the
  requested `repoId` (Q4 virtual-repo coherence).
- `asOf` is the catalog watermark (max `last_seen_at`), reusable as `since`.

### Bulk enumeration / projection-export (FR-5, FR-6, FR-7, FR-10, FR-12)
```
GET /api/v1/catalog/versions?repoId={slug}&since={cursor}&page={n}
-> 200 { rows: [{ repoId, appId, version, promotionStage }], nextCursor, asOf }
```
- Deterministic, stably ordered (FR-10) so projection rebuilds are diffable.
- `since` omitted -> full rebuild; `since={asOf}` -> deltas only (FR-6),
  driven by `last_seen_at`.

## Cross-cutting (both PRs)
- **Auth (FR-8/FR-9):** read-only; 401 unauthenticated. Reuses
  `REPOFABRIC_PUBLISHER_TOKEN` per the bolt-on posture (Q6).
- **Batch (Q8):** add a `POST /api/v1/catalog/presence:batch` variant once
  ConfigFabric confirms its resolver fans out, to avoid author-time round-trips.
- **Tests:** presence hit/miss, unknown-repo clean-negative, coherence
  (dev-only version not satisfying a main query), enumeration full vs `since`
  delta, stage passthrough, and case-folding.
