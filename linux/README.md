# RepoFabric, Linux container

A single-container deployment of RepoFabric for any Docker host. Browser-managed over HTTPS. Operated by RingoSystems Heavy Industries.

Per-platform guides live under [`admin/static/docs/`](admin/static/docs/): plain Linux, UNRAID, Synology, TrueNAS, and Portainer. The admin UI also serves them at `/docs/` once it is up.

## Status

0.9.0 is the current release, and the first public release. It carries client-side bandwidth optimization (PeerDist LAN peer-caching, Intune endpoint configuration, a savings dashboard) behind a default-off kill switch, the integrated ConfigFabric sidecar (flag-gated, off by default), repo-aware version retention, and the first phase of a cross-fabric hardening program. Containers behave identically to a clean install until an operator opts into PeerDist via service.yaml. See [`../CHANGELOG.md`](../CHANGELOG.md) for the full history.

## Prerequisites on the host

1. The companion stack from `../deploy/docker-compose.yml` is running: `repofabric-gitea` and `repofabric-rewinged`.
2. The `repofabric` docker network exists. `deploy/bootstrap.sh` creates it.
3. An Entra app registration in your tenant with a redirect URI matching `${REPOFABRIC_ADMIN_PUBLIC_URL}/auth/callback` and the `User.Read`, `User.Read.All`, `Group.Read.All` Microsoft Graph permissions granted.
4. A reverse proxy (Nginx Proxy Manager is the documented default) configured to route `https://winget.<domain>/admin/` and `https://winget.<domain>/setup/` to `repofabric-linux:8086`, and `https://installers.<domain>/` to `repofabric-linux:8091`, and `https://winget.<domain>/api/` to `repofabric-rewinged:8080`.

## Install walkthrough

```bash
# 1. Run the bootstrap (creates dirs + docker network + starter .env).
#    Add --appdata-root <path> to put the manifest tree + installer cache on separate bulk storage.
bash deploy/bootstrap.sh /var/lib/repofabric

# 2. Edit the .env file generated by bootstrap.sh and fill in:
#    REPOFABRIC_ADMIN_PUBLIC_URL, REPOFABRIC_ENTRA_*, REPOFABRIC_GITEA_PAT,
#    and optional SMTP credentials.
nano deploy/.env

# 3. Bring up the companion containers
docker compose -f deploy/docker-compose.yml up -d gitea rewinged

# 4. Create the winget-manifests repo in Gitea via http://<host>:3030,
#    generate a write-scoped PAT, paste it into .env as
#    REPOFABRIC_GITEA_PAT.

# 5. Build and bring up RepoFabric
docker compose -f linux/docker-compose.yml up -d --build

# 6. Read the setup token from logs
docker logs repofabric-linux 2>&1 | grep -A2 "Setup token"
```

Then open `https://winget.<your-domain>/setup/`, paste the token, and walk the wizard. On Save the wizard writes `service.yaml` and `solution.yaml` under `/var/lib/repofabric/config/`, deletes the token file, marks setup complete, and renders a Setup complete page with a manual link to `/admin/`.

## Operator surfaces

- **Service Configuration tab:** worker pool size, sync schedule, preferred architectures, retention defaults, log level.
- **Solution Configuration tab:** Entra tenant and allowed users/groups, target endpoints, SMTP relay.
- **Subscriptions tab:** managed (upstream-tracked), custom-published, and untracked apps in one combined view.
- **Custom publish wizard:** full WinGet manifest schema v1.6.0 surface (Platform, MinimumOSVersion, InstallModes, InstallerSwitches, ExpectedReturnCodes, AppsAndFeaturesEntries, dependencies, multi-installer, multi-locale, nested installer support).
- **Virtual repos:** a CRUD modal under Settings. Each virtual repo gets its own Gitea repo and its own Rewinged container, spawned via the docker-driver over the bind-mounted host `docker.sock`.
- **Promotion:** cross-repo manifest copy with one click from the source repo.
- **Activity tab:** unified `run` UNION `admin_event` feed (sync runs, custom publishes, retention cleanups, drift events, restores).
- **Banner:** drift detection surfaces uninvited Gitea commits as a banner on the Catalog tab.
- **Backup tab:** archive snapshots and the DR drill button (`Test-RfDisasterRecovery`).

## Secrets

The `.env` file at the path you chose in step 1 (default `/etc/repofabric/.env` if you ran with `--appdata-root` matching the system convention) holds every secret. The wizard never asks for plaintext credentials in solution.yaml; the wizard's `gitea_pat` field is the one exception and the value is also overridable from `REPOFABRIC_GITEA_PAT`.

The six required envs:

- `REPOFABRIC_ADMIN_PUBLIC_URL`
- `REPOFABRIC_SESSION_SECRET` (32+ random chars; bootstrap fills this automatically)
- `REPOFABRIC_ENTRA_TENANT_ID`
- `REPOFABRIC_ENTRA_CLIENT_ID`
- `REPOFABRIC_ENTRA_CLIENT_SECRET`
- `REPOFABRIC_GITEA_PAT`

Optional SMTP envs:

- `REPOFABRIC_SMTP_USERNAME` and `REPOFABRIC_SMTP_PASSWORD` for SMTP auth.
- Host, port, from, to are persisted to `solution.yaml` by the wizard so they show up in the admin UI.

## Installer integrity

Installer downloads are verified by SHA-256 only. There is no signature posture beyond the hash, so operators who require signature verification should layer external verification.
